üîç What is CORS?

Cross-Origin Resource Sharing (CORS) is a security mechanism implemented by browsers to control how resources on a web server can be requested from another domain (origin). It is designed to prevent malicious websites from reading sensitive data from another domain without permission.

A ‚Äúcross-origin‚Äù request occurs when a web application attempts to request resources from a different domain, protocol, or port than the one it originated from.

‚ö†Ô∏è Why CORS is Important

Browsers restrict cross-origin HTTP requests initiated from scripts for security reasons ‚Äî known as the Same-Origin Policy. Without CORS, web applications couldn't safely share resources across domains, limiting integration capabilities.

CORS provides a controlled way to relax these restrictions while maintaining security by allowing servers to specify who can access their resources and how.

‚öôÔ∏è How CORS Works

When a browser makes a cross-origin HTTP request, it sends an Origin header indicating the requesting domain.

The server can respond with specific headers such as:

  • Access-Control-Allow-Origin: Specifies which origins are permitted.
  • Access-Control-Allow-Methods: Lists allowed HTTP methods (GET, POST, etc.).
  • Access-Control-Allow-Headers: Specifies allowed request headers.
  • Access-Control-Allow-Credentials: Indicates whether cookies or credentials are allowed.

If the response headers permit the request‚Äôs origin and method, the browser allows the response to be accessed by the client-side script.

üîÑ Simple vs Preflight Requests

Simple requests are cross-origin requests that meet certain criteria (e.g., methods like GET or POST with standard headers). These requests are sent directly.

Preflight requests are sent by the browser automatically for more complex requests (e.g., methods like PUT, DELETE, or custom headers). This is an HTTP OPTIONS request sent before the actual request to check if the cross-origin call is allowed.

üìã Common CORS Headers

  • Access-Control-Allow-Origin: Specifies allowed origin(s). Can be a specific domain or * to allow all.
  • Access-Control-Allow-Methods: Allowed HTTP methods for cross-origin requests.
  • Access-Control-Allow-Headers: Allowed headers that the client can send.
  • Access-Control-Allow-Credentials: When true, cookies or HTTP authentication information can be sent.
  • Access-Control-Max-Age: Indicates how long the results of a preflight request can be cached.

üõ†Ô∏è How to Enable CORS in APIs

Most web frameworks offer built-in or middleware solutions to enable and configure CORS easily.

Examples:

  • Node.js (Express): Use the cors middleware to configure allowed origins, methods, headers, and credentials.
  • Spring Boot: Use @CrossOrigin annotation or configure CORS globally.
  • ASP.NET Core: Configure CORS in the startup with policies specifying allowed origins and headers.

üîí Security Considerations

Improper CORS configuration can lead to security vulnerabilities:

  • Setting Access-Control-Allow-Origin to * while allowing credentials exposes sensitive data.
  • Allowing all methods and headers can increase attack surface.
  • Always whitelist specific trusted origins and limit methods/headers.

üêû Troubleshooting CORS Issues

Common CORS errors include:

  • No ‚ÄòAccess-Control-Allow-Origin‚Äô header: The server did not send CORS headers.
  • Credentials error: Using withCredentials on client but server disallows credentials.
  • Preflight failure: Server rejects or does not respond properly to OPTIONS preflight request.

To fix, check server CORS configuration, validate headers, and test with tools like Postman or browser devtools.

üëî Senior/Lead Needs to Know

  • Understand the balance between enabling cross-origin requests for business needs and protecting API security.
  • Ensure CORS policies are aligned with overall API security strategies and are regularly audited.
  • Guide team to implement strict origin whitelisting rather than permissive * wildcard usage.
  • Architect APIs to minimize sensitive operations exposed via cross-origin calls or require additional authentication.
  • Monitor and log cross-origin requests to detect suspicious or unauthorized access patterns.
  • Educate developers on preflight request impacts and optimize APIs to reduce unnecessary preflight overhead.
  • Integrate CORS configuration as part of CI/CD pipelines and infrastructure as code to maintain consistency.
